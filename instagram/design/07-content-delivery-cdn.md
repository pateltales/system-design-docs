## 1. Meta's CDN Infrastructure

Instagram uses **Meta's own CDN infrastructure** — a combination of owned Edge PoPs (Points of Presence) and direct peering agreements. [VERIFIED — Facebook Engineering blog posts, 2014 and 2017]

Meta does NOT primarily rely on third-party CDNs (Akamai, CloudFront). At Meta's scale, operating their own CDN is more cost-effective and provides better control over performance.

### Architecture

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  End User    │────>│  Edge PoP (L1)   │────>│  Origin DC       │
│  (mobile app)│     │  (dozens globally)│     │  (Meta data      │
│              │     │  Hot content      │     │   centers)        │
│              │     │  cached locally   │     │  Haystack/f4     │
└──────────────┘     └──────────────────┘     └──────────────────┘
                           │
                     ┌─────▼────────────┐
                     │ Regional Cache   │
                     │ (L2, mid-tier)   │
                     │ Warm content     │
                     └──────────────────┘
```

**Edge PoPs:**
- Deployed at IXPs (Internet Exchange Points) and carrier-neutral facilities worldwide
- Estimated dozens of PoPs on 6 continents [APPROXIMATE — exact count not publicly disclosed]
- Cache hot content (recent posts from popular accounts, trending Reels)
- DNS-based routing directs users to the nearest PoP

**Regional Caches (L2):**
- Mid-tier caches within or near Meta's data centers
- Store warm content (older posts, moderate-popularity content)
- Absorb cache misses from Edge PoPs before hitting origin

**Origin Storage:**
- Haystack for hot photo storage (one disk I/O per read)
- f4 for warm photo storage (erasure-coded, space-efficient)
- All media ultimately lives here

## 2. Image Serving Optimization

Instagram is image-first. Optimizing image delivery has outsized impact on performance and bandwidth costs.

### Multiple Resolutions

Every photo is stored in 4 resolution variants (generated by the media processing pipeline):

| Variant | Width | Use Case | Typical Size |
|---|---|---|---|
| Thumbnail | 150×150 | Profile grid, search results | ~5-10 KB |
| Small | 320px | Feed on small screens, previews | ~20-40 KB |
| Medium | 640px | Feed on standard phones | ~50-100 KB |
| Large | 1080px | Full-screen view, HD displays | ~100-300 KB |

The client requests the appropriate variant based on:
- Device screen resolution
- Current network quality (WiFi vs cellular)
- Display context (grid thumbnail vs feed post vs full-screen)

### Modern Image Formats

Instagram serves images in the most efficient format supported by the client:

| Format | Savings vs JPEG | Support | When Used |
|---|---|---|---|
| **JPEG** | Baseline | Universal | Fallback for all clients |
| **WebP** | ~25-30% smaller | Android, Chrome | Default for Android clients |
| **AVIF** | ~50% smaller | Newer iOS/Android | Where supported |

**Content negotiation:** The client sends an `Accept` header indicating supported formats. The CDN/server returns the most efficient supported format.

**Impact at scale:** If Instagram serves ~1 billion image loads/day and WebP saves 25% bandwidth per image:
- Average image: ~80KB (JPEG) → ~60KB (WebP) = ~20KB savings per load
- 1B loads × 20KB = **~20 TB of bandwidth saved per day**

### Progressive JPEG

Instagram uses progressive JPEG encoding:
- Image loads in layers: blurry preview first → progressively sharpens
- Better perceived performance than baseline JPEG (which loads top-to-bottom)
- User sees a recognizable image almost immediately, even on slow connections

### Blurhash Placeholders

While the real image loads, the client renders a **blurhash** — a compact (~20-30 byte) string encoding a blurred color preview.

```
Before image loads:          After image loads:
┌────────────────────┐      ┌────────────────────┐
│                    │      │                    │
│   [colored blur    │      │   [actual photo    │
│    placeholder]    │  →   │    rendered]       │
│                    │      │                    │
└────────────────────┘      └────────────────────┘
```

**Why blurhash instead of a tiny thumbnail?**
- Blurhash is ~30 bytes, included inline in the API response (no extra HTTP request)
- A tiny JPEG thumbnail would be ~500+ bytes and require a separate fetch
- Blurhash provides a smooth visual transition — eliminates the jarring "blank white box" that users see while images load

## 3. Video Serving (HLS)

Feed videos and Reels are served via **HLS (HTTP Live Streaming)** — Apple's adaptive bitrate streaming protocol.

### How HLS Works

```
1. Video is split into 2-4 second segments
2. Each segment exists at multiple quality levels (360p, 480p, 720p, 1080p)
3. A master playlist (M3U8) lists all quality levels
4. The client downloads segments one at a time, choosing quality per-segment
5. If bandwidth drops → next segment is lower quality (seamless switch)
```

**Playlist structure:**
```
#EXTM3U
#EXT-X-STREAM-INF:BANDWIDTH=500000,RESOLUTION=640x360
360p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=1000000,RESOLUTION=854x480
480p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=2500000,RESOLUTION=1280x720
720p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=5000000,RESOLUTION=1920x1080
1080p/playlist.m3u8
```

**Why HLS (not DASH)?**
- Instagram is mobile-first. HLS is native on iOS (Apple's protocol).
- Android supports HLS via ExoPlayer (Google's media player library).
- DASH (MPEG-DASH) is more common for web/TV platforms (Netflix uses DASH).
- Using HLS simplifies the client — one protocol for both iOS and Android.

### Reels-Specific Video Delivery

Reels are full-screen vertical videos that auto-play. The delivery strategy is optimized for:

1. **Instant playback**: First segment downloads ASAP → play immediately
2. **Prefetching**: Buffer next 2-3 Reels while current one plays
3. **Quality ramp-up**: Start at lower quality for instant start → ramp up as buffer fills
4. **Auto-play**: Videos play automatically as user scrolls — no user-initiated load

## 4. Caching Strategy

Instagram uses **reactive caching** — content is cached at the edge on first request, NOT proactively pushed.

### Why Reactive (Not Proactive)?

| Approach | When It Works | When It Fails |
|---|---|---|
| **Proactive push** (Netflix) | Curated catalog, predictable demand | Billions of UGC items, unpredictable access |
| **Reactive caching** (Instagram) | UGC, long-tail distribution | First request has higher latency (cache miss) |

Instagram has **100M+ uploads/day** with a long-tail access pattern — most photos are viewed primarily by the uploader's followers in the first few hours. Proactively pushing all content to all Edge PoPs would waste enormous bandwidth. Reactive caching naturally caches popular content and lets rare content serve from origin.

### Cache Tiers

```
┌──────────────────────────────────────────────────────────────────┐
│ Client-side cache (app memory + disk)                            │
│   • Feed images cached locally                                    │
│   • Reduces API calls and image downloads on app re-open          │
│   • LRU eviction based on device storage                          │
│   Hit rate: Very high for recently viewed content                 │
├──────────────────────────────────────────────────────────────────┤
│ CDN Edge cache (L1 — Edge PoPs)                                  │
│   • Hot content cached at edge, close to users                    │
│   • Recent posts from popular accounts, trending Reels            │
│   • Hit rate: >90% for hot content                                │
│   • TTL: Hours to days (varies by content popularity)             │
├──────────────────────────────────────────────────────────────────┤
│ Regional cache (L2 — mid-tier)                                   │
│   • Warm content cached at regional level                         │
│   • Older posts, moderate-popularity accounts                     │
│   • Absorbs misses from Edge before hitting origin                │
│   • Hit rate: ~70-80% for remaining requests                      │
├──────────────────────────────────────────────────────────────────┤
│ Origin storage (Haystack/f4)                                     │
│   • All content stored permanently                                │
│   • Only ~5-10% of requests reach origin                          │
│   • Cold content (old posts, rarely viewed)                       │
└──────────────────────────────────────────────────────────────────┘
```

### Cache Invalidation

When media is deleted or modified:
- **CDN purge**: Send invalidation to all Edge PoPs that may have the content cached
- **URL-based**: Instagram uses content-hash-based URLs — changing the content changes the URL, so old cached versions naturally become unreachable
- **TTL-based**: Even without explicit purge, cached content expires after TTL

## 5. URL Structure & Security

Instagram's CDN URLs contain embedded metadata:

```
https://cdn.instagram.com/{size}/{hash}/{format}/{signature}/{filename}

Example:
https://scontent-lax3-2.cdninstagram.com/v/t51.2885-15/
e35/s1080x1080/
12345678_abc_n.jpg
?_nc_ht=scontent-lax3-2.cdninstagram.com
&_nc_cat=1
&_nc_ohc=signature_hash
&edm=ABmJApABAAAA
&ccb=7-5
&oh=token_signature
&oe=expiration_timestamp
```

**Components:**
- **Edge server identifier** (`scontent-lax3-2`) — which PoP is serving
- **Size variant** (`s1080x1080`) — which resolution
- **Content hash** — for cache busting (content changes → URL changes)
- **Signature/token** — prevents hotlinking and unauthorized access
- **Expiration** — URLs have a TTL; expired URLs return 403

**Security considerations:**
- URLs are signed — cannot be tampered with to access other sizes/content
- Expiration prevents indefinite hotlinking from external sites
- CDN validates the signature before serving content

## 6. Contrasts

### Instagram vs Netflix — CDN Strategy

| Dimension | Instagram | Netflix |
|---|---|---|
| **CDN ownership** | Meta-owned Edge PoPs | Netflix-owned Open Connect Appliances (OCAs) |
| **Caching model** | Reactive (cache on first request) | Proactive push (pre-position before request) |
| **Why?** | UGC: 100M+ uploads/day, unpredictable access | Curated: ~tens of thousands of titles, predictable |
| **Content type** | Mostly images (KB), some video (MB) | Exclusively video (GB) |
| **Per-item bandwidth** | Low (50-300KB per image) | High (GB per stream) |
| **Total items** | Billions (every photo ever posted) | Tens of thousands (curated catalog) |
| **Cache hit ratio** | >90% for hot content | ~95%+ (nearly everything pre-cached) |
| **ISP integration** | Standard peering | OCAs installed inside ISP networks |

**Key insight:** Netflix's curated catalog makes proactive push viable — you can predict what users will watch and pre-position it. Instagram's UGC volume makes proactive push impossible — you can't push 100M new photos to every PoP every day. Reactive caching naturally adapts to actual demand.

### Instagram vs Twitter — CDN Load

| Dimension | Instagram | Twitter |
|---|---|---|
| **Primary content** | Media-first (photos/videos) | Text-first (280 chars) |
| **CDN load per feed item** | Heavy (50-300KB per image download) | Light (text is inline in API response) |
| **Bottleneck** | CDN bandwidth and edge cache capacity | API throughput and timeline assembly |
| **Image optimization** | Critical (WebP, AVIF, blurhash, progressive JPEG) | Secondary (images are optional) |
| **Feed load network cost** | ~1-5 MB per page (5-20 images) | ~50-200 KB per page (mostly text) |

**Key insight:** Instagram's CDN must handle 10-100x the media bandwidth of Twitter per feed load. This is why Instagram invests heavily in image format optimization (WebP, AVIF) and multi-resolution serving — savings compound at billions of image loads per day.
